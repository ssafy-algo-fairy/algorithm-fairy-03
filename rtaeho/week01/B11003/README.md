### 📌 문제 정보

- **번호:** 11003  
- **제목:** 최솟값 찾기  
- **난이도:** Gold 1 
- **분류:** 자료구조, 덱(Deque), 슬라이딩 윈도우

---

### 💡 접근 방식

> 길이 L의 슬라이딩 윈도우에서 각 구간의 최솟값을 구하는 문제입니다.  
> 매번 윈도우의 최솟값을 계산하면 시간 초과가 발생하므로, **덱을 이용해 항상 오름차순을 유지하는 방식**으로 해결했습니다.

---

### 🔹 1단계 – 덱에 값과 인덱스 저장

- 덱에는 `[인덱스, 값]` 형태로 저장
- 덱의 앞(front)에는 항상 **현재 윈도우의 최솟값**이 위치하도록 유지

---

### 🔹 2단계 – 덱 정렬 유지

- 현재 값이 덱의 마지막 값보다 작으면  
- 해당 값은 앞으로 최솟값이 될 가능성이 없으므로 제거

```java
while (!deque.isEmpty() && cur < deque.peekLast()[1]) {
    deque.removeLast();
}
```

---

### 🔹 3단계 – 윈도우 범위 벗어난 값 제거

- 덱의 첫 번째 원소가 윈도우 범위 `[i-L+1, i]`를 벗어나면 제거

```java
if (deque.peekFirst()[0] <= i - L) {
    deque.removeFirst();
}
```

---

### 🔹 4단계 – 현재 윈도우의 최솟값 출력

덱의 첫 번째 값이 항상 현재 구간의 최솟값입니다.

```java
sb.append(deque.peekFirst()[1]).append(" ");
```

---

### 💻 핵심 로직

#### 덱에서 큰 값 제거

```java
while (!deque.isEmpty() && cur < deque.peekLast()[1]) {
    deque.removeLast();
}
```

#### 윈도우 범위 벗어난 값 제거

```java
if (deque.peekFirst()[0] <= i - L) {
    deque.removeFirst();
}
```

---

### ⏳ 복잡도 분석

- **시간 복잡도:** `O(N)`
    - 각 원소는 덱에 한 번 들어가고 한 번 제거됨

- **공간 복잡도:** `O(L)`
    - 덱에는 최대 `L`개의 원소만 유지

---

### ⚠️ 어려웠던 점

- 처음에는 **PriorityQueue**를 사용해 구간의 최솟값을 관리하려고 했지만,  
  범위를 벗어난 값을 제거하는 과정 때문에 시간 초과가 발생했습니다.
- 그래서 덱을 사용해 **현재 구간에서 필요 없는 값들을 미리 제거하는 방식**으로 해결했습니다.
