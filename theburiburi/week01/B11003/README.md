# 🔍 최솟값 찾기 (11003) 풀이 발표

---

## 🌳 문제 핵심

> $N$개의 수 $A_1, A_2, ..., A_N$과 $L$이 주어질 때, $D_i = min(A_{i-L+1}, ..., A_i)$를 구하는 문제입니다.
> 데이터의 개수 $N$이 최대 **5,000,000**이므로, 일반적인 정렬이나 우선순위 큐($O(N \log N)$)를 사용하면 **시간 초과**가 발생합니다.
> 따라서 **덱(Deque)을 활용하여 $O(N)$에 해결하는 슬라이딩 윈도우 알고리즘**이 필수적입니다.

---

## 💡 핵심 알고리즘 : 슬라이딩 윈도우 (Sliding Window) / 덱 (Deque)

```text
덱을 활용한 최적화 = 모든 원소를 한 번씩만 넣고 빼며 최솟값 유지
  └─ 덱(Deque)에는 값의 '인덱스'를 저장하여 범위 체크 용이하게 구성
      └─ 새로운 원소(arr[i])가 들어올 때마다 다음 두 단계 수행
            ├─ 뒤에서 제거: 덱에 있는 기존 값이 현재 값보다 크면 제거 (오름차순 유지)
            ├─ 앞에서 제거: 덱의 맨 앞 인덱스가 현재 윈도우 범위(i-L)를 벗어나면 제거
            └─ 결과 추출: 덱의 맨 앞(peekFirst)에는 항상 현재 구간의 최솟값 인덱스가 위치
```

```
// 1. 새로운 값(arr[i])이 들어올 때, 덱 뒤쪽에서 현재 값보다 큰 값들은 모두 제거
// 현재 값보다 큰 값들은 앞으로 최솟값이 될 가능성이 전혀 없기 때문 (오름차순 정렬 상태 유지)
while (!dq.isEmpty() && arr[dq.peekLast()] > arr[i]) {
    dq.removeLast();
}

// 2. 현재 인덱스 삽입
dq.addLast(i);

// 3. 윈도우 범위를 벗어난 인덱스 제거 (i - L 보다 작거나 같은 인덱스)
if (dq.peekFirst() <= i - L) {
    dq.removeFirst();
}

// 4. 덱의 맨 앞 인덱스가 가리키는 값이 현재 구간의 정답
sb.append(arr[dq.peekFirst()]).append(" ");
```

항목,내용
탐색 방식,슬라이딩 윈도우 (Sliding Window)
자료구조,덱 (Deque - Monotonic Queue)
시간 복잡도,O(N) (각 원소당 최대 1번 삽입/삭제)
공간 복잡도,O(N) (전체 배열 및 덱 저장 공간)

🔑 Key Point:**단조 큐(Monotonic Queue)**의 원리가 핵심입니다. 덱 내부를 항상 오름차순 상태로 유지함으로써, 최솟값을 찾기 위해 전체 범위를 뒤지는 것이 아니라 peekFirst()를 통해 $O(1)$만에 즉시 값을 꺼낼 수 있게 됩니다. 또한, 값 자체가 아닌 인덱스를 저장함으로써 윈도우가 오른쪽으로 이동할 때 유효 기간이 지난 데이터를 쉽게 걸러낼 수 있습니다.


🎯 결론 및 배울 점
시간 복잡도의 한계 돌파: $N$이 500만인 경우 $O(N \log N)$ 알고리즘도 위험할 수 있다는 점을 인지하고, 선형 시간($O(N)$) 내에 처리할 수 있는 자료구조를 고민하는 능력을 길렀습니다.

효율적인 입출력: 데이터 양이 방대할 때는 Scanner보다는 BufferedReader, System.out.println보다는 StringBuilder나 BufferedWriter를 사용하는 것이 성능에 결정적인 영향을 미친다는 것을 실습했습니다.

덱의 양방향성 활용: 상황에 따라 덱의 앞과 뒤에서 유연하게 데이터를 삭제하는 로직을 통해 큐나 스택보다 강력한 자료구조 활용법을 익혔습니다.
