# 백준 33156 - 구간이 이븐하지 않아요

## 1. 문제 요약

주어진 수열 $A$에서 연속된 부분 수열 $A[l, r]$을 선택했을 때, 이를 절반으로 나눈 두 수열 $A[l, m]$과 $A[m+1, r]$이 서로 **이븐(Even)**한 관계인지 확인하여 그 최대 길이를 구하는 문제입니다. 

* **이븐(Even)의 정의**: 두 수열의 길이가 같고, 수열 내 원소의 위치를 바꾸는 연산을 통해 서로 같아질 수 있는 상태를 의미합니다. 즉, **두 수열을 구성하는 숫자의 종류와 각 숫자의 개수(빈도수)가 완벽히 일치**해야 합니다. 


* **제한 사항**: 수열의 길이 $N$은 최대 5,000이며, 원소의 값은 $10^9$까지 가능합니다. 


* **목표**: 조건을 만족하는 $A[l, r]$ 중 가장 긴 길이를 출력하며, 존재하지 않으면 0을 출력합니다. 



---

## 2. 사용 로직 및 핵심 알고리즘

이 문제는 수열의 구성 요소가 같은지 확인하는 **빈도수 체크**와 **브루트포스(완전 탐색)**를 결합하여 해결합니다. 

* **최대 길이부터 하향식 탐색**: 문제에서 요구하는 것은 '최대 길이'이므로, 가능한 최대 짝수 길이($N$ 또는 $N-1$)부터 시작하여 2씩 줄여가며 탐색합니다. 조건을 만족하는 길이를 찾는 즉시 프로그램을 종료하여 효율성을 높였습니다. 


* **슬라이딩 윈도우 기반 빈도수 관리**:
* 고정된 길이 `len`에 대해 시작 지점을 이동시키며 부분 수열을 검사합니다. 


* `HashMap`을 사용하여 앞부분($m1$)과 뒷부분($m2$)의 원소 개수를 관리합니다. 


* 윈도우가 한 칸 이동할 때마다 모든 원소를 다시 세는 대신, 새로 들어오는 원소와 나가는 원소만 `compute`와 `merge`를 통해 업데이트합니다. 




* **이븐 조건 판별**: `m1.equals(m2)`를 호출하여 두 맵의 내용(Key-Value 쌍)이 동일한지 비교함으로써 두 수열이 이븐한지 판별합니다. 



---

## 3. 코드 하이라이트

제출된 코드 중 슬라이딩 윈도우를 통해 빈도수 맵을 동적으로 갱신하는 핵심 부분입니다. 

```java
while (start + len < n) {
    // 윈도우 이동에 따른 m1 갱신 (맨 앞 제거, 중간 경계값 추가)
    m1.compute(num[start], (k, v) -> v == null || v == 1 ? null : v - 1);
    m1.merge(num[start + len / 2], 1, (old, v) -> old + v);
    
    // 윈도우 이동에 따른 m2 갱신 (중간 경계값 제거, 맨 뒤 추가)
    m2.compute(num[start + (len / 2)], (k, v) -> v == null || v == 1 ? null : v - 1);
    m2.merge(num[start + len], 1, (old, v) -> old + v);
    
    // 두 해시맵의 구성 요소가 동일한지 비교
    if (m1.equals(m2))
        break mainLoop;
    start++;
}
```

---

## 4. 시간 복잡도 분석

* **시간 복잡도**: $O(N^3)$ (최악의 경우) 


* 바깥쪽 루프(`len`)는 최대 $N/2$번 반복됩니다. 


* 안쪽 슬라이딩 윈도우 루프는 약 $N$번 반복됩니다. 


* 매 이동마다 `HashMap.equals()`가 발생하며, 이는 맵에 저장된 고유 원소의 개수($K$)에 비례합니다 ($O(K)$). 


* $N=5,000$일 때 최악의 경우 연산량이 많을 수 있으나, 문제 데이터의 특성과 `equals()`의 조기 종료 특성 덕분에 주어진 시간 제한 내에 통과가 가능합니다. 





---

## 5. 학습 포인트 및 개선 사항

* **데이터 구조 활용**: 대규모 범위의 원소 값($10^9$)을 처리하기 위해 배열 인덱싱 대신 `HashMap`을 사용하여 메모리를 효율적으로 관리했습니다. 


* **최적화 제안 (차이 추적 방식)**: 현재는 매번 두 맵 전체를 비교하고 있습니다. 만약 두 맵의 빈도수 차이를 계산하는 변수 `diffCount`를 도입하여, 갱신 시점에 `diffCount`가 0이 되는지 확인하는 **차이 추적(Difference Tracking)** 방식을 사용한다면, 비교 연산을 $O(1)$로 줄여 전체 복잡도를 $O(N^2)$까지 최적화할 수 있습니다. 



---

**추가 분석**: 현재 코드 구조상 $N=5,000$ 조건에서 $O(N^3)$은 다소 위험할 수 있으나, 슬라이딩 윈도우의 효율적인 갱신 덕분에 성공적으로 동작합니다.
